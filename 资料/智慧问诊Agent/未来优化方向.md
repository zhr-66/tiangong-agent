# 智慧问诊 Agent — 未来优化方向

> 本文档汇总自教程 `资料/智慧问诊Agent/完整功能实现` 中各 Step 的"未来优化方向"章节。
> 基本需求已全部实现，以下为后续可选优化项，按模块归类，未排期。

---

## 一、症状标准化（Step 2 / symptom_normalizer.py）

1. **Redis 热词缓存**
   - 在 Redis 中缓存 `{症状原词: 标准词}` 映射，命中缓存后跳过 LLM，节省约 500ms。
   - 查到标准词的直接用，查不到的走 Milvus 语义兜底。

2. **动态相似度阈值**
   - `SIMILARITY_THRESHOLD` 当前固定为 0.85。
   - 可按疾病领域动态调整（罕见病领域调低到 0.80，常见病领域保持 0.85）。

---

## 二、Milvus 症状索引（Step 3 / scripts/init_symptom_index.py）

1. **增量更新**
   - 当前为全量重建脚本。可监听 Neo4j 变更事件（或定时对比），只对新增症状做 upsert，不重建全量索引。
   - 注意唯一性原则：以症状名为主键，upsert 天然去重。

2. **多语言支持**
   - 未来若需支持英文症状，可在同一 collection 里加 `lang` 字段做过滤。

---

## 三、Neo4j 图谱查询（Step 4 / neo4j_queries.py）

1. **Cypher 并发执行**
   - `enrich_candidate_details` 中四个 Cypher 查询当前串行执行。
   - 可改为 `asyncio.gather` 并发执行提升速度（类比 Java JUC 的 `CompletableFuture.allOf`）。

2. **Cypher 合并**
   - 四个 enrich 查询可合并成一个多 `OPTIONAL MATCH` 的 Cypher，减少网络往返。

3. **高频疾病详情 Redis 预热**
   - 对高频疾病（症状、科室、检查），可在应用启动时预热到 Redis，查询时直接命中缓存。

---

## 四、置信度计算（Step 5 / confidence.py）

1. **易感人群加权接入**
   - `+0.05` 易感人群匹配当前已注释，等 `Disease.easy_get` 字段的 NLP 解析接入后启用。
   - 当前为简化实现，实际匹配逻辑需结合 `easy_get` 字段内容（年龄段/性别）。

---

## 五、Prompt 模板（Step 6 / prompts.py）

1. **PARSE_ANSWER_PROMPT 改用结构化输出**
   - 当前用 JSON 字符串解析，可改用 `with_structured_output` 配合 Pydantic 模型，比解析 JSON 字符串更稳定。

2. **分人群 Prompt 变体**
   - 为不同年龄段的患者（老人/儿童）准备不同语气的 Prompt 变体，提升体验。

---

## 六、LangGraph 主体（Step 8 / graph.py）

1. **interrupt 机制替代手动 END**
   - 当前 `clarify` 和 `ask_symptoms` 后接 `END` 等待用户回复。
   - 可用 LangGraph 的 `interrupt_before` / `interrupt_after` 机制替代，更优雅地实现"暂停等待用户输入"。

2. **图编译缓存**
   - `build_inquiry_graph` 每次调用都重新编译图。
   - 可改为模块级缓存（`@lru_cache`），提升性能。

3. **问诊状态持久化**
   - 当前 InquiryState 存在自定义 Redis 键中。
   - 可加入 LangGraph 的官方 checkpointer（Redis），让问诊状态持久化，用户关闭页面后再回来可以继续上次的问诊。

---

## 七、Inquiry Worker Agent（Step 9 / workers/inquiry_agent.py）

1. **真实挂号工具接入**
   - 当前 `tools = []` 为占位，`handle_handoff` 用 LLM 生成模拟挂号数据。
   - 未来接入真实挂号系统时，添加：预约挂号工具、问诊单生成工具、排班查询工具。

---

## 八、call_inquiry_agent 工具（Step 10 / worker_tools.py）

1. **db_session 依赖注入**
   - `build_inquiry_deps` 中的 `db_session` 目前为 `None`。
   - 需要接入 FastAPI 的 `AsyncSession` 依赖注入，让工具内也能正确使用数据库会话。

2. **TTL 活跃自动续期**
   - 当前 TTL 固定 1 小时。
   - 可在每次用户有新消息时重置 TTL，实现"活跃自动续期"。

---

## 九、FastAPI 路由（Step 11 / api/routers/chat.py）

1. **InquiryGraph 真正逐 token 流式**
   - 当前问诊进行中时，InquiryGraph 内部多次调用 LLM，结果整体推送，体验略有延迟。
   - 可在 `node_ask_symptoms`、`node_conclude` 等节点内部使用 `llm.astream()` 并通过 `asyncio.Queue` 向外传递 token，实现真正的逐字流式。

2. **JWT 鉴权中间件**
   - 当前 `user_id` 由客户端手动传。
   - 加入 JWT 鉴权中间件，从 Token 中自动提取 `user_id`，不需要客户端手动传。

---

## 优先级建议（仅供参考）

| 优先级 | 优化项 | 价值 |
|--------|--------|------|
| 高 | JWT 鉴权中间件 | 安全性，生产必备 |
| 高 | build_inquiry_deps 接入 AsyncSession 依赖注入 | 修复工具内 DB 会话为 None 的隐患 |
| 高 | 真实挂号工具接入 | 业务闭环 |
| 中 | Redis 热词缓存 | 性能，节省 LLM 调用 |
| 中 | Cypher 并发/合并 | 性能，减少网络往返 |
| 中 | 图编译 lru_cache | 性能 |
| 中 | interrupt 机制 | 代码可维护性 |
| 低 | 易感人群加权 | 召回准确率 |
| 低 | PARSE_ANSWER 结构化输出 | 稳定性（当前已有降级） |
| 低 | 分人群 Prompt 变体 | 体验 |
| 低 | 多语言支持 | 扩展性，非当前需求 |
